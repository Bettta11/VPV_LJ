# Инструкция агенту: проект `lj-vdw`

## 0. Общая цель проекта

Нужно создать компактный вычислительный проект для микро-исследовательской работы по общей физике. Тема работы:

**Проверка применимости уравнения Ван-дер-Ваальса к Lennard-Jones-флюиду по данным молекулярно-динамического моделирования.**

Физическая идея проекта:

1. Берём модельное вещество: частицы взаимодействуют потенциалом Леннарда—Джонса.
2. В приведённых единицах строим численные изотермы (P(\rho,T)).
3. Подбираем параметры (a,b) уравнения Ван-дер-Ваальса по MD-данным.
4. Сравниваем MD-изотермы с Ван-дер-Ваальсом.
5. Отдельно делаем визуальные симуляции для демонстрации фазового поведения, но не смешиваем их с основными численными данными.

Главный результат проекта — не “красивая симуляция”, а таблица EOS-точек и графики сравнения:

* (P_{\mathrm{MD}}(\rho,T));
* fitted (P_{\mathrm{VdW}}(\rho,T));
* отклонения модели Ван-дер-Ваальса от MD;
* качественные финальные профили плотности по высоте, чтобы видеть характер конечного состояния.

Проект должен быть достаточно простым, чтобы пользователь мог понимать его структуру и при необходимости объяснить, что происходит в каждом файле. Не нужно строить промышленный research framework.

## 1. Важные физические и методические решения

### 1.1. Единицы и модель

Использовать приведённые LJ-единицы:

```text
sigma = 1
epsilon = 1
mass = 1
kB = 1
```

Потенциал:

```text
U(r) = 4 epsilon [(sigma/r)^12 - (sigma/r)^6]
```

Использовать 3D-модель. Основной движок — OpenMM.

Допустимые варианты cutoff:

```text
rcut = 2.5
```

Можно реализовать LJTS или LJFS. Если неясно, выбрать более простой устойчивый вариант и явно зафиксировать его в конфиге и логах. Важно, чтобы один и тот же cutoff/protocol использовался во всех EOS-прогонах.

### 1.2. Три независимых подпроекта

Проект делится на три режима:

```text
eos    — основные численные данные для изотерм и fit Ван-дер-Ваальса
debug  — короткие тестовые запуски для проверки устойчивости и визуальной диагностики
visual — красивые траектории/видео, не используемые как источник EOS-данных
```

Эти режимы могут использовать общие функции из `core.py`, но хранят данные отдельно и имеют разную политику сохранения.

### 1.3. Без slab на первом этапе

Не реализовывать slab-геометрию на первом этапе.

Причина: пользователь пока не хочет отдельный искусственный протокол с центральной жидкой областью, recentering slab и извлечением отдельных плотностей фаз. Основная работа сейчас строится по изотермам, а не по независимому измерению бинодали.

Slab можно будет добавить позже как отдельное расширение, если данных из EOS окажется недостаточно.

### 1.4. Без гравитации в EOS

В `eos`-режиме гравитацию не использовать:

```yaml
external_field:
  type: none
  g: 0.0
```

Причина: EOS должна соответствовать однородной периодической системе без внешнего поля. Гравитация меняет равновесное состояние и загрязняет сравнение с (P(\rho,T)).

Гравитацию можно поддержать в `core.py` как опциональную возможность, но использовать её только в `debug` или `visual`.

### 1.5. Не сохранять лишние данные

В `eos` не сохранять траектории и не создавать отдельную папку на каждую точку. Все точки одной EOS-серии должны попадать в одну аккуратную таблицу.

Не создавать `eos_morphology.csv`. Он лишний.

Финальные профили плотности сохранять только как количество частиц по пространственным бинам, а не как “обработанную морфологию”.

## 2. Требуемая структура проекта

Создать следующую компактную структуру:

```text
lj-vdw/
├── cloud_runner.ipynb
├── core.py
├── eos.py
├── debug.py
├── visual.py
│
├── configs/
│   ├── eos.yaml
│   ├── debug.yaml
│   └── visual.yaml
│
├── data/
│   ├── eos/
│   ├── debug/
│   └── visual/
│
└── report_assets/
    ├── figures/
    └── videos/
```

Назначение файлов:

```text
core.py
    Общие функции: чтение YAML, создание OpenMM-системы, LJ-потенциал,
    начальные условия, термостат, внешний потенциал, запуск MD-шагов,
    извлечение observables.

eos.py
    Массовые EOS-прогоны, таблица eos_points.csv, финальные профили,
    fit Ван-дер-Ваальса, графики изотерм.

debug.py
    Маленькие тестовые запуски с подробной диагностикой и короткой траекторией.

visual.py
    Генерация красивых траекторий/видео. Эти данные не должны использоваться
    как основной источник EOS.

cloud_runner.ipynb
    Управляющий ноутбук. В нём не должно быть длинной логики.
    Он только импортирует функции, запускает конфиги и показывает результаты.
```

Не создавать на старте:

```text
pyproject.toml
src/ljvdw/...
scripts/
сложную систему вложенных пакетов
много отдельных notebooks
HDF5/Zarr-хранилища
автоматическую промышленную систему задач
```

Проект должен оставаться простым.

## 3. Формат данных

### 3.1. `eos`

Директория одной EOS-серии:

```text
data/eos/eos_001/
├── config.yaml
├── eos_points.csv
├── eos_final_profiles.csv
├── eos_trace_sample.csv       # опционально
├── vdw_fit.json               # после анализа
├── figures/
│   ├── eos_isotherms.png
│   ├── vdw_fit.png
│   └── profile_overview.png
└── log.txt
```

В `eos` не создавать отдельную папку на каждую точку ((T,\rho)). Все точки равноправно складываются в `eos_points.csv`.

#### `eos_points.csv`

Одна строка = один прогон при заданных параметрах.

Минимальные колонки:

```text
run_id
T_target
T_mean
rho_target
rho_mean
N
Lx
Ly
Lz
dt
equil_steps
prod_steps
seed
P_mean
P_std
P_sem
U_mean
U_std
U_sem
K_mean
E_mean
status
notes
```

`status` использовать реально:

```text
ok
not_equilibrated
unstable_temperature
phase_separated
failed
```

По умолчанию все точки считаются равноправными, пока диагностика не показывает обратное.

#### `eos_final_profiles.csv`

Сохранять только количество частиц по бинам для финального состояния или небольшого финального окна.

Формат:

```text
run_id
bin
z_min
z_max
z_center
count
```

Именно `count`, а не обработанную плотность. Плотность можно восстановить на этапе анализа, зная размеры ящика и площадь сечения.

Не сохранять:

```text
eos_morphology.csv
полные координаты
полные скорости
trajectory.dcd
positions.npy
velocities.npy
```

Профиль нужен только как лёгкая диагностика характера конечного состояния: однородность, кластеризация, возможное фазовое разделение. Он не является точным методом извлечения плотностей фаз.

#### `eos_trace_sample.csv`

Опциональный файл. Можно выключить в production.

Если включён, сохранять редкий временной ряд:

```text
run_id
step
T
P
U
K
E
```

Интервал:

```text
раз в 5000–10000 шагов
```

Нужен только для проверки равновесия и объяснения плохих точек. Не должен раздувать проект.

### 3.2. `debug`

Директория debug-серии:

```text
data/debug/debug_001/
├── config.yaml
├── debug_runs.csv
├── state_trace.csv
├── trajectory.dcd
├── topology.pdb
├── profiles.csv
├── figures/
│   ├── energy_trace.png
│   ├── temperature_trace.png
│   ├── pressure_trace.png
│   ├── final_profile.png
│   └── snapshot.png
└── log.txt
```

Debug может сохранять больше данных, потому что размеры маленькие. Эти данные не идут в итоговый анализ.

В debug нужно проверять:

* OpenMM запускается;
* температура удерживается термостатом;
* энергия не взрывается;
* при малой плотности (P \approx \rho T);
* частицы не накладываются;
* начальные условия корректны;
* визуализация открывает траекторию.

Ограничить размер debug-траектории:

```yaml
trajectory:
  enabled: true
  interval: 200
  max_frames: 200
```

### 3.3. `visual`

Директория visual-серии:

```text
data/visual/visual_001/
├── config.yaml
├── state_trace.csv
├── trajectory.dcd
├── topology.pdb
├── profiles_time.csv          # только если нужен side-panel с rho(z)
├── videos/
│   └── demo.mp4
├── figures/
│   ├── final_snapshot.png
│   ├── profile_final.png
│   └── pv_curve.png
└── log.txt
```

Visual-режим предназначен для красивой демонстрации. Здесь можно использовать:

* гравитацию;
* сжатие;
* стенки;
* цвет по скорости;
* цвет по локальной плотности;
* side-panel с профилем плотности;
* time-lapse.

Но visual-данные не использовать для fit Ван-дер-Ваальса.

## 5. Этапы реализации

Работать строго по этапам. Не забегать вперёд. После каждого этапа проверять, что код запускается и создаёт ожидаемые файлы.

### Этап 1. Скелет проекта и общий OpenMM-core

Сделать:

1. Создать структуру директорий.
2. Реализовать `core.py`.
3. Реализовать чтение YAML.
4. Реализовать создание OpenMM-системы с LJ-потенциалом в приведённых единицах.
5. Реализовать начальную расстановку частиц:

   * для малых плотностей можно случайно с минимальным расстоянием;
   * для плотных состояний использовать решётку/регулярное размещение, чтобы частицы не накладывались.
6. Реализовать Langevin-термостат.
7. Реализовать один короткий debug-запуск.
8. Проверить, что создаются:

   * `config.yaml`;
   * `log.txt`;
   * `state_trace.csv`;
   * короткая `trajectory.dcd`.

Не реализовывать на этом этапе:

* fit Ван-дер-Ваальса;
* видео;
* EOS sweep;
* сложный анализ;
* slab.

Критерий завершения этапа:

```text
python/debug или cloud_runner.ipynb запускает маленькую LJ-систему,
температура не взрывается,
state_trace.csv создаётся,
trajectory.dcd открывается.
```

### Этап 2. EOS-подпроект

Сделать:

1. Реализовать `eos.py`.
2. Реализовать проход по сетке `temperatures × densities × seeds`.
3. Все результаты складывать в одну директорию `data/eos/eos_001/`.
4. Не создавать отдельные папки для каждой точки.
5. Заполнять `eos_points.csv`.
6. Сохранять `eos_final_profiles.csv` как counts по z-бинам для финального окна.
7. Не сохранять траектории в EOS.
8. Сделать первичные графики:

   * (P(\rho)) для разных (T);
   * (U(\rho)) для разных (T);
   * обзор финальных профилей.

Критерий завершения этапа:

```text
configs/eos.yaml запускает маленькую EOS-серию,
eos_points.csv содержит все точки,
eos_final_profiles.csv содержит только counts по бинам,
из eos_points.csv строится график изотерм.
```

### Этап 3. Анализ Ван-дер-Ваальса

Сделать:

1. Добавить в `eos.py` или отдельный блок в `cloud_runner.ipynb` fit параметров (a,b).

2. Использовать модель:

   P = rho*T/(1 - b*rho) - a*rho^2

3. Сохранять результат в `vdw_fit.json`.

4. Строить графики:

   * MD-изотермы;
   * fitted VdW-кривые;
   * residuals;
   * сравнение по температурным сериям.

5. Fit-region должен задаваться в конфиге или явно в коде анализа, не быть магически зашитым без комментария.

Не нужно на этом этапе:

* точно строить бинодаль;
* извлекать отдельные плотности жидкой и газовой фаз;
* считать скрытую теплоту;
* реализовывать Maxwell construction, если это усложняет первый working version.

Критерий завершения этапа:

```text
по eos_points.csv строится сравнение MD и VdW,
vdw_fit.json содержит a,b и описание fit-region,
графики сохраняются в data/eos/eos_001/figures/.
```

### Этап 4. Debug/visual-подпроекты и cloud_runner

Сделать:

1. Довести `debug.py` до удобного локального режима:

   * короткая траектория;
   * графики T(t), P(t), E(t);
   * snapshot.
2. Довести `visual.py`:

   * траектория;
   * рендер короткого preview-видео;
   * optional side-panel с профилем плотности;
   * сжатие или другой демонстрационный протокол.
3. Оформить `cloud_runner.ipynb` как главный управляющий ноутбук:

   * установка зависимостей;
   * проверка OpenMM platform;
   * запуск debug;
   * запуск eos;
   * запуск анализа;
   * запуск visual.
4. В ноутбуке не должно быть длинной реализации. Только импорт функций и orchestration.

Критерий завершения этапа:

```text
проект можно запустить из cloud_runner.ipynb,
debug работает быстро,
eos создаёт таблицы,
analysis строит графики,
visual создаёт mp4 или хотя бы набор preview-кадров.
```

## 6. Требования к стилю реализации

1. Не усложнять архитектуру без необходимости.
2. Не создавать много вложенных пакетов.
3. Не плодить файлы и артефакты.
4. В `eos` не сохранять траектории.
5. В `eos` не создавать папку на каждую точку.
6. В `eos_final_profiles.csv` сохранять только количество частиц по бинам.
7. Не добавлять `eos_morphology.csv`.
8. Не реализовывать slab на первом этапе.
9. Не использовать гравитацию в `eos`.
10. Любая новая функция должна быть привязана к одному из трёх режимов: `eos`, `debug`, `visual`.

## 7. Минимальный ожидаемый API

В идеале из ноутбука должно быть возможно сделать так:

```python
from debug import run_debug
from eos import run_eos, fit_vdw, plot_eos_results
from visual import run_visual, render_video

debug_dir = run_debug("configs/debug.yaml")

eos_dir = run_eos("configs/eos.yaml")
fit_vdw(eos_dir)
plot_eos_results(eos_dir)

visual_dir = run_visual("configs/visual.yaml")
render_video(visual_dir)
```

Если реализация требует немного другой API, это допустимо, но она должна оставаться такой же простой по смыслу.

## 8. Что считать успехом проекта

Минимально успешная версия:

```text
1. OpenMM запускает LJ-систему.
2. debug-режим показывает устойчивость расчёта.
3. eos-режим создаёт eos_points.csv и eos_final_profiles.csv.
4. По eos_points.csv строятся изотермы.
5. Ван-дер-Ваальс fit'ится по выбранной области.
6. Строится график MD vs VdW.
7. visual-режим создаёт хотя бы короткую красивую траекторию или видео.
```

Итоговая работа должна позволить пользователю сказать:

> Я взял микроскопическую LJ-модель вещества, численно построил её изотермы, подобрал к ним уравнение Ван-дер-Ваальса и проверил, где это макроскопическое приближение работает, а где начинает расходиться с молекулярной динамикой.
