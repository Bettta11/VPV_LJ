# Задание: ноутбук с визуализацией изотермического сжатия LJ-флюида

Нужно создать в корне проекта самостоятельный Kaggle-ноутбук:

```text
lj_compression_movie.ipynb
```

Ноутбук должен:

1. провести один непрерывный MD-прогон изотермического сжатия Lennard–Jones-флюида при (T=0.8);
2. сохранить численные данные и координаты кадров;
3. отдельно от симуляции отрендерить трёхмерный ролик;
4. показать справа стационарную изотерму (P(\rho)), динамическую траекторию сжатия и текущие параметры системы;
5. сохранить итоговый MP4.

Главное требование — простота и прозрачность. Не создавать отдельные модули, классы, конфигурационные файлы, сложные системы логирования и архитектурные слои. Ноутбук должен последовательно читаться сверху вниз и запускаться через `Run all`.

## Научный смысл ролика

Ролик представляет не последовательность равновесных состояний, а **неравновесное изотермическое сжатие при конечной скорости**.

Это должно быть явно указано:

* в первой Markdown-ячейке;
* в заголовке диаграммы;
* в подписи внутри ролика.

Стационарная MD-изотерма (P(\rho)) используется как фон и ориентир. Динамическая траектория сжатия может отклоняться от неё из-за запаздывания структурной релаксации, метастабильности и гистерезиса. Не пытаться искусственно совместить эти две кривые.

Ролик должен качественно показать три стадии:

1. разреженный газ;
2. кластеризацию и прохождение немонотонной области, включая отрицательное давление;
3. переход к плотной жидкоподобной фазе.

Перед написанием ноутбука просмотреть существующие рабочие ноутбуки проекта и переиспользовать уже проверенные способы:

* создания LJ-системы;
* задания периодических граничных условий;
* создания начальных координат;
* настройки OpenMM;
* вычисления давления по вириальной формуле;
* перевода величин в приведённые LJ-единицы;
* загрузки CSV с EOS-данными.

Не переписывать реализацию давления по памяти, если в проекте уже есть рабочий вариант.

Не изменять существующие рабочие ноутбуки без необходимости.

## Базовая физическая модель

Использовать те же reduced LJ units, что и в основном проекте:

```python
sigma = 1.0
epsilon = 1.0
mass = 1.0
kB = 1.0

N = 512
temperature = 0.8

rho_start = 0.05
rho_end = 0.85

rcut = 2.5
dt = 0.002
```

Система:

```text
трёхмерная кубическая коробка;
периодические граничные условия;
Lennard–Jones-взаимодействие;
NVT-динамика;
термостат Ланжевена;
изотропное сжатие с сохранением кубической формы коробки.
```

Использовать `LangevinMiddleIntegrator`, если он совместим с существующей реализацией проекта.

Начальная коробка определяется из:

[
L=\left(\frac{N}{\rho}\right)^{1/3}.
]

Начальные координаты должны создаваться тем же безопасным способом, что и в основном ноутбуке: регулярная решётка без перекрытий, при необходимости с небольшим возмущением.

## Использование Kaggle и GPU

Ноутбук будет запускаться в Kaggle с двумя Tesla T4.

Одна траектория сжатия является последовательной, поэтому не нужно искусственно распараллеливать её на две GPU.

По умолчанию использовать одну GPU:

```python
DeviceIndex = "0"
Precision = "mixed"
```

Не распределять систему из 512 частиц между двумя T4 без явной причины: для такой маленькой системы межкарточные накладные расходы могут быть больше выигрыша.

В начале ноутбука:

* вывести версию Python;
* вывести версию OpenMM;
* выполнить `nvidia-smi`;
* вывести список платформ OpenMM;
* проверить создание настоящего CUDA Context;
* завершить выполнение с ясной ошибкой, если CUDA не работает.

Вторую T4 можно оставить неиспользуемой. Это предпочтительнее усложнения ноутбука.

## Схема сжатия

Сжатие должно быть ступенчатым, но визуально выглядеть почти непрерывным.

На каждой стадии:

1. вычислить следующую целевую плотность;
2. вычислить новую длину коробки;
3. изотропно масштабировать координаты частиц вместе с коробкой;
4. установить новые периодические векторы;
5. некоторое число шагов эволюционировать при фиксированной новой коробке;
6. сохранять измерения и координаты кадров;
7. перейти к следующей стадии.

Масштабирование координат должно сохранять дробные координаты:

[
\mathbf r_i^{,\mathrm{new}}
===========================

\mathbf r_i^{,\mathrm{old}}
\frac{L_{\mathrm{new}}}{L_{\mathrm{old}}}.
]

После изменения коробки координаты должны быть корректно возвращены внутрь периодической ячейки.

Скорости при каждом сжатии не пересоздавать. Термостат должен поддерживать температуру и отводить тепло, возникающее при компрессии.

Перед началом сжатия выполнить:

```python
initial_equil_steps = 20_000
```

при постоянной начальной плотности `rho_start`.

## Три участка процесса

Плотности и число стадий должны находиться в одной конфигурационной ячейке и легко изменяться.

Начальный вариант:

```python
COMPRESSION_SEGMENTS = [
    {
        "name": "Разреженный газ",
        "rho_from": 0.05,
        "rho_to": 0.15,
        "n_stages": 20,
        "steps_per_stage": 500,
    },
    {
        "name": "Кластеризация и неустойчивая область",
        "rho_from": 0.15,
        "rho_to": 0.55,
        "n_stages": 60,
        "steps_per_stage": 1_500,
    },
    {
        "name": "Переход к плотной жидкости",
        "rho_from": 0.55,
        "rho_to": 0.85,
        "n_stages": 30,
        "steps_per_stage": 750,
    },
]
```

После достижения конечной плотности выполнить дополнительную выдержку:

```python
final_hold_steps = 10_000
```

без дальнейшего изменения коробки.

Границы `0.15` и `0.55` являются начальными настройками. Перед финальным запуском сопоставить их с фактической стационарной изотермой (T=0.8). При необходимости изменить так, чтобы второй участок покрывал наблюдаемую немонотонную область и отрицательные давления.

Не делать отдельную сложную систему автоматического определения этих границ.

## Частота измерений и кадров

Разделить численные измерения и визуальные кадры.

Рекомендуемые параметры:

Цель — получить порядка 400–500 кадров.

## Сглаживание давления

В CSV сохранять мгновенные значения давления.

Для ролика использовать только центрированное скользящее среднее, рассчитанное после завершения симуляции.

Начальный параметр:

```python
pressure_smoothing_window_steps = 2_500
```

Число измерений в окне вычислить из `measurement_interval`.

Не использовать фильтр, который сильно смещает момент изменения давления относительно структуры системы.

Для краёв ряда использовать разумное неполное окно, чтобы не получать пропуски.

На экране показывать:

```text
T = 0.800
ρ = ...
P̄ = ...
```

где (P̄) — сглаженное давление.

Мгновенное давление в кадре не показывать.

### Левая часть

Трёхмерная сцена с частицами и сжимающейся кубической коробкой.

Использовать PyVista:

* off-screen rendering;
* фиксированную камеру;
* неподвижный угол обзора;
* одинаковый размер маркеров частиц;
* отображение точек как освещённых сфер;
* тонкие рёбра кубической коробки;
* отсутствие осей, координатной сетки и лишних подписей;
* спокойный однотонный фон.

Не создавать 512 отдельных полигональных сфер. Использовать эффективное отображение точек как сфер, например через `render_points_as_spheres`.

Координаты для визуализации центрировать:

[
\mathbf r_i^{\mathrm{view}}
===========================

\mathbf r_i-\frac{L}{2}.
]

Камера должна оставаться фиксированной относительно начальной коробки. Границы сцены не должны автоматически масштабироваться под текущий размер ящика. Визуально коробка должна действительно уменьшаться.

Начальная установка камеры примерно:

```text
elevation: 20–25°
azimuth: 35–45°
focal point: центр коробки
```

Не добавлять движение камеры.

### Правая часть

Использовать Matplotlib.

Показывать:

* стационарную изотерму (P(\rho)) при (T=0.8);
* горизонтальную линию (P=0);
* уже пройденную часть динамической траектории;
* текущую точку ((\rho,P_{\mathrm{smooth}}));
* текущие значения (T), (\rho), (P_{\mathrm{smooth}});
* название текущего участка процесса.

Оси должны оставаться неизменными весь ролик.

Стационарная изотерма и динамическая траектория должны иметь разные подписи:

```text
Стационарная MD-изотерма
Динамическое сжатие
```

Заголовок:

```text
Неравновесное изотермическое сжатие LJ-флюида
```

Не перегружать правую часть дополнительными графиками. Не показывать одновременно (U(\rho)), (Z(\rho)) и другие величины.

## Сборка итогового кадра

Для каждого кадра:

1. PyVista рендерит левую 3D-панель в изображение;
2. Matplotlib рендерит правую панель в изображение;
3. изображения объединяются через Pillow или NumPy;
4. итоговый кадр сохраняется во временную директорию либо непосредственно передаётся в ffmpeg.

Использовать фиксированный размер обеих панелей, чтобы кадр не дёргался.

После успешной сборки MP4 временные PNG-кадры можно удалить или оставить по конфигурационному флагу:

```python
KEEP_RENDERED_FRAMES = False
```

Кодирование выполнять через ffmpeg с совместимым форматом:

```text
codec: libx264
pixel format: yuv420p
fps: 20
```

В начале и в конце ролика можно удержать первый и последний кадр примерно по одной секунде.

## Синхронизация данных и кадров

Каждому сохранённому набору координат сопоставить:

* ближайшее по времени сглаженное давление;
* текущую плотность;
* текущую длину коробки;
* текущий участок процесса.

Динамическая траектория справа должна строиться только до текущего момента ролика, а не показываться целиком заранее.

Текущий маркер должен двигаться по динамической траектории.
